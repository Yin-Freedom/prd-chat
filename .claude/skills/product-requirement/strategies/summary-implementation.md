# 摘要策略实现方案

## 一、问题分析

### 挑战

Skill文件（SKILL.md）是静态的Markdown文档，它定义了工作流程，但：
- 无法自动执行代码逻辑
- 无法自动检测token数量
- 无法主动触发压缩

### 解决思路

通过在skill中**定义摘要规则和数据结构**，让LLM在对话过程中**主动执行摘要**。

---

## 二、实现方案

### 方案A：阶段完成摘要

每个阶段完成后，生成结构化摘要，替代详细对话历史。

**在SKILL.md中添加**：

```markdown
## 阶段摘要机制

每个阶段完成确认后，生成结构化摘要：

### 摘要格式

```json
{
  "phase": "阶段名称",
  "status": "completed",
  "collected": {
    // 本阶段收集的数据
  },
  "decisions": [
    "关键决策1",
    "关键决策2"
  ],
  "pending": [
    "待确认项1"
  ]
}
```

### 示例

阶段1完成后：

```json
{
  "phase": "产品背景",
  "status": "completed",
  "collected": {
    "product": {
      "name": "工单管理系统",
      "description": "企业内部工单提交和处理平台",
      "targetUsers": ["普通员工", "部门经理", "IT管理员"],
      "coreValue": "提高工单处理效率，实现流程可追溯"
    }
  },
  "decisions": [],
  "pending": []
}
```

### 执行时机

阶段确认后，**在继续下一阶段前**，输出摘要并说明：
> "已记录产品背景信息，继续下一阶段..."
```

### 方案B：对话历史压缩

定义对话压缩规则，在长对话中主动压缩。

**在SKILL.md中添加**：

```markdown
## 对话压缩规则

### 压缩触发条件

当对话满足以下任一条件时，主动压缩历史：

1. 用户进行了3次以上修改请求
2. 单阶段对话超过10轮
3. 用户明确要求"回顾"或"总结"

### 压缩方法

将详细对话历史替换为结构化摘要：

**压缩前（多轮对话）**：
```
用户: 这个系统包含哪些模块？
Claude: 识别到3个模块：用户管理、订单管理、报表统计
用户: 用户管理有哪些功能？
Claude: 新增用户、查询用户、修改用户、删除用户
用户: 用户名有什么要求？
Claude: 用户名需要唯一，6-20位...
...
```

**压缩后（摘要）**：
```json
{
  "currentPhase": "功能模块",
  "collected": {
    "modules": ["用户管理", "订单管理", "报表统计"],
    "用户管理": {
      "functions": ["新增", "查询", "修改", "删除"],
      "constraints": ["用户名唯一", "6-20位"]
    }
  },
  "inProgress": "订单管理模块详情",
  "history": "已收集产品背景、用户角色、用户管理模块"
}
```
```

### 方案C：数据状态管理

定义一个"已收集数据"结构，在对话中持续维护。

**在SKILL.md中添加**：

```markdown
## 数据状态管理

在对话过程中，维护一个结构化的"已收集数据"对象：

### 状态对象

```json
{
  "meta": {
    "productName": "",
    "currentPhase": 1,
    "totalPhases": 5
  },
  "product": null,
  "roles": null,
  "modules": null,
  "flows": null,
  "nonFunctional": null
}
```

### 状态更新规则

1. **阶段完成时**：更新对应字段，标记为已完成
2. **用户修改时**：更新对应字段，记录修改历史
3. **每次回复前**：检查当前状态，确保不遗漏已收集信息

### 状态展示

当用户请求回顾时，展示当前状态：

```
📊 当前进度

✅ 产品背景（已完成）
✅ 用户角色（已完成）
🔄 功能模块（进行中 - 2/3模块）
⬜ 业务流程
⬜ 非功能需求

已收集模块：
- 用户管理（完成）
- 订单管理（进行中）
```
```

## 三、推荐实现

将上述方案整合到SKILL.md中，创建完整的摘要策略。

### 实现要点

1. **在SKILL.md开头定义状态对象模板**
2. **定义每个阶段完成后的摘要行为**
3. **定义修改历史的管理规则**
4. **定义进度回顾机制**

### SKILL.md添加内容

```markdown
## 📋 数据状态管理

### 状态对象模板

在对话过程中维护此结构：

```json
{
  "meta": {
    "productName": "",
    "currentPhase": 1,
    "progress": "0/5"
  },
  "data": {
    "product": null,
    "roles": null,
    "modules": null,
    "flows": null,
    "nonFunctional": null
  },
  "history": []
}
```

### 阶段完成行为

每完成一个阶段：
1. 更新 `data` 中对应字段
2. 更新 `meta.progress`
3. 简要说明已记录，不重复展示详细内容

示例输出：
> ✅ 已记录产品背景。进入阶段2：用户角色...

### 修改历史管理

当用户要求修改已收集内容时：
1. 更新对应数据
2. 在 `history` 中添加修改记录
3. 展示修改后的简要摘要

### 进度回顾

当用户输入"回顾"、"进度"、"已完成"等关键词时，展示：

```
📊 当前进度: 3/5 阶段

✅ 产品背景
✅ 用户角色
🔄 功能模块 (2/3模块完成)
⬜ 业务流程
⬜ 非功能需求
```
```

## 四、执行流程

```
用户触发skill
    ↓
初始化状态对象
    ↓
阶段1问答 → 用户确认 → 更新状态 → 摘要输出
    ↓
阶段2问答 → 用户确认 → 更新状态 → 摘要输出
    ↓
...
    ↓
输出生成 → 加载对应prompt → 生成文档/原型
```

## 五、示例对话

```
Claude: 你好！我是产品需求助手。
        请告诉我产品名称是什么？

用户: 工单管理系统

Claude: 请描述这个产品是什么？

用户: 企业内部工单提交和处理平台

...（更多问答）

Claude: 产品背景确认完成：
        - 名称: 工单管理系统
        - 简介: 企业内部工单提交和处理平台
        - 目标用户: 普通员工、部门经理、IT管理员
        - 核心价值: 提高效率、流程可追溯

        ✅ 已记录产品背景。进入阶段2：用户角色...

        这个系统涉及哪些用户角色？

用户: 回顾一下进度

Claude: 📊 当前进度: 1/5 阶段

        ✅ 产品背景（已完成）
        ⬜ 用户角色
        ⬜ 功能模块
        ⬜ 业务流程
        ⬜ 非功能需求

        继续收集用户角色信息吗？
```

## 六、Token节省效果

| 场景 | 无摘要 | 有摘要 | 节省 |
|------|--------|--------|------|
| 完成5阶段 | ~15,000 tokens | ~8,000 tokens | ~47% |
| 多次修改 | +5,000 tokens/次 | +500 tokens/次 | ~90% |
| 进度回顾 | 重复历史 | 结构化摘要 | ~80% |
