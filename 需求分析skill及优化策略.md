基于《Lost in the Middle: How Language Models Use Long Contexts》及相关研究了解到，越长的上下文导致模型性能显著下降。

### 分块策略

因此，使用分块策略按需加载prompt和template，充分利用LLM的智能。

```
启动 → 加载: SKILL.md + quick-reference.md (~4,500 tokens)
阶段1(产品背景) → 加载: phase1-product.md (~520 tokens)
阶段2(用户角色) → 加载: phase2-roles.md (~600 tokens)
阶段3(功能模块) → 加载: phase3-modules.md (~1,200 tokens)
阶段4(业务流程) → 加载: phase4-flows.md (~780 tokens)
阶段5(非功能需求) → 加载: phase5-nonfunctional.md (~890 tokens)
生成文档 → 加载: document-generation.md (~5,000 tokens)
生成原型 → 加载: prototype-generation.md (~5,800 tokens)
```



### 摘要策略

**核心思想**

用"当前状态"替代"历史过程"——只记录结果，不重复过程。

**问题**

Skill文件（SKILL.md）是静态的Markdown文档，它定义了工作流程，但：

- 无法自动执行代码逻辑
- 无法自动检测token数量
- 无法主动触发压缩
- **无法真正删除历史对话**（上下文由对话平台管理）

**解决方案：阶段切换/clear提示**

在SKILL.md中各阶段确认后添加`/clear`提示，引导用户主动清空历史：

```
> ✅ 已记录产品背景。
> 💡 **提示**：输入 `/clear` 可清空对话历史，减少上下文占用后继续下一阶段
> 进入阶段2：用户角色...
```

**实现位置**：

- 阶段完成行为示例
- 阶段1-5的确认输出

**效果**：
- 用户主动控制上下文大小
- 真正减少LLM的上下文负担
- 配合状态对象实现无缝继续

**思路**

通过在skill中定义摘要规则和数据结构，让LLM在对话过程中主动执行摘要。

定义一个状态对象

```
{
    "meta": { "productName": "", "currentPhase": 1, "progress": "0/5" },
    "data": { "product": null, "roles": null, "modules": null, "flows": null, "nonFunctional": null }
}
```

每完成一个阶段后：
  1. 更新 data 中对应字段
  2. 更新 meta.progress
  3. 简要说明已记录，不重复展示详细对话历史

通过模拟两个Agent对比有无摘要策略的效果：

单阶段Token消耗对比（阶段1：产品背景）

| 指标 | 无摘要策略 | 有摘要策略 | 节省比例 |
|------|-----------|-----------|---------|
| 字符数 | ~5,171 | ~3,320 | ~36% |
| Token数 | ~4,041 | ~2,200 | ~46% |
| 输出风格 | 完整复述对话过程 | 结构化状态记录 |

完整流程Token消耗对比（5个阶段完成）

| 指标 | 无摘要策略 | 有摘要策略 | 节省比例 |
|------|-----------|-----------|---------|
| 累计Token | ~20,000+ | ~10,000 | ~50% |
| 对话轮次保留 | 完整历史 | 仅当前状态 |
| 修改历史 | 重复展示 | 简要摘要 |

**无摘要策略（阶段1完成后的上下文）**：

```
用户: 这个产品叫什么名字？
Claude: 请告诉我这个产品的名称是什么？
用户: 工单管理系统
Claude: 好的，产品名称是"工单管理系统"。接下来...
用户: 这个产品是做什么的？
Claude: 请用一两句话描述这个产品...
用户: 企业内部工单提交和处理平台
Claude: 已记录：工单管理系统，企业内部工单提交和处理平台...
（完整保留所有对话历史）
```
**有摘要策略（阶段1完成后的上下文）**：

```json
{
  "meta": { "productName": "工单管理系统", "currentPhase": 2, "progress": "1/5" },
  "data": {
    "product": {
      "name": "工单管理系统",
      "description": "企业内部工单提交和处理平台",
      "targetUsers": "待收集",
      "coreValue": "待收集"
    }
  }
}
```

### 上下文压缩

**目的**

1. **提升模型性能**：研究表明，LLM在长上下文中的性能会显著下降，尤其是需要从上下文中检索信息时，"中间位置"的信息最容易被忽略（Lost in the Middle现象）

2. **降低Token成本**：减少每次请求的Token数量，直接降低API调用成本

3. **提高响应速度**：更少的上下文意味着更快的推理速度


**已有的实现方法**

**方法1：滑动窗口压缩**

只保留最近N轮对话，更早的对话通过摘要保留关键信息：

```
┌─────────────────────────────────────┐
│ 摘要区（固定保留）                    │
│ - 产品名称、核心描述                  │
│ - 已完成的阶段状态                    │
│ - 关键决策记录                       │
├─────────────────────────────────────┤
│ 活跃区（滑动窗口，保留最近3-5轮）      │
│ - 当前阶段的对话                     │
│ - 待确认的问题                       │
└─────────────────────────────────────┘
```

**方法2：分层压缩策略**

| 层级 | 内容类型 | 压缩方式 | 保留比例 |
|------|---------|---------|---------|
| L1 | 状态对象 | JSON结构化存储 | 100% |
| L2 | 当前阶段对话 | 原样保留 | 100% |
| L3 | 已完成阶段 | 摘要压缩 | ~20% |
| L4 | 历史迭代 | 仅保留最终结果 | ~5% |

**实际使用方法**

由于 / 开头斜杠命令是 ClaudeCode 系统内置的，不能通过自定义 tool 来拓展。因此，考虑增加阶段提示，在任务的不同阶段设置提示，告诉用户手动调用 /clear + /skill。同时，在skill中增加启动流程，进入skill时先检查记忆状态，提示当前进度。
