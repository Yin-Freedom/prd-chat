## 需求分析应用开发

### 分块策略

**目的**

基于《Lost in the Middle》研究，长上下文会导致模型性能下降。通过分块按需加载，保持上下文精简。

**实现方式**

在SKILL.md中定义prompt模板引用，LLM按阶段自动读取对应文件：

```
启动 → SKILL.md + quick-reference.md (~4,500 tokens)
阶段1 → phase1-product.md (~1,500 tokens)
阶段2 → phase2-roles.md (~1,500 tokens)
阶段3 → phase3-modules.md (~3,000 tokens)
阶段4 → phase4-flows.md (~1,500 tokens)
阶段5 → phase5-nonfunctional.md (~1,000 tokens)
输出生成 → document-generation.md (~5,000 tokens) / prototype-generation.md (~5,800 tokens)
```

**效果**

| 方式 | 单阶段上下文 | 全流程累计 |
|------|-------------|-----------|
| 一次性加载全部prompt | ~20,000+ tokens | 持续增长 |
| 分块按需加载 | ~6,000 tokens | 稳定可控 |



### 摘要策略

**核心思想**

用"当前状态"替代"历史过程"——只记录结果，不重复过程。

**问题**

Skill文件（SKILL.md）是静态的Markdown文档，它定义了工作流程，但：

- 无法自动执行代码逻辑
- 无法自动检测token数量
- 无法主动触发压缩
- **无法真正删除历史对话**（上下文由对话平台管理）

**解决方案：持久化摘要 + 阶段切换/clear提示**

在SKILL.md中各阶段确认后添加`/clear`提示，引导用户主动清空历史：

```
> ✅ 已记录产品背景。
> 💡 **提示**：输入 `/clear` 可清空对话历史，减少上下文占用后继续下一阶段
> 进入阶段2：用户角色...
```

**实现位置**：

- 阶段完成行为示例
- 阶段1-5的确认输出

**效果**：
- 用户主动控制上下文大小
- 真正减少LLM的上下文负担
- 配合状态对象实现无缝继续

**思路**

通过在skill中定义摘要规则和数据结构，让LLM在对话过程中主动执行摘要。

定义一个状态对象

```
{
    "meta": { "productName": "", "currentPhase": 1, "progress": "0/5" },
    "data": { "product": null, "roles": null, "modules": null, "flows": null, "nonFunctional": null }
}
```

每完成一个阶段后：
  1. 更新 data 中对应字段
  2. 更新 meta.progress
  3. 简要说明已记录，不重复展示详细对话历史

通过模拟两个Agent对比有无摘要策略的效果：

单阶段Token消耗对比（阶段1：产品背景）

| 指标 | 无摘要策略 | 有摘要策略 | 节省比例 |
|------|-----------|-----------|---------|
| 字符数 | ~5,171 | ~3,320 | ~36% |
| Token数 | ~4,041 | ~2,200 | ~46% |
| 输出风格 | 完整复述对话过程 | 结构化状态记录 |

完整流程Token消耗对比（5个阶段完成）

| 指标 | 无摘要策略 | 有摘要策略 | 节省比例 |
|------|-----------|-----------|---------|
| 累计Token | ~20,000+ | ~10,000 | ~50% |
| 对话轮次保留 | 完整历史 | 仅当前状态 |
| 修改历史 | 重复展示 | 简要摘要 |

**无摘要策略（阶段1完成后的上下文）**：

```
用户: 这个产品叫什么名字？
Claude: 请告诉我这个产品的名称是什么？
用户: 工单管理系统
Claude: 好的，产品名称是"工单管理系统"。接下来...
用户: 这个产品是做什么的？
Claude: 请用一两句话描述这个产品...
用户: 企业内部工单提交和处理平台
Claude: 已记录：工单管理系统，企业内部工单提交和处理平台...
（完整保留所有对话历史）
```
**有摘要策略（阶段1完成后的上下文）**：

```json
{
  "meta": { "productName": "工单管理系统", "currentPhase": 2, "progress": "1/5" },
  "data": {
    "product": {
      "name": "工单管理系统",
      "description": "企业内部工单提交和处理平台",
      "targetUsers": "待收集",
      "coreValue": "待收集"
    }
  }
}
```



### 记忆机制

**核心思想**

用"状态文件"替代"对话历史"——将已收集的数据持久化存储，实现跨会话恢复和摘要策略。

**解决的问题**

| 问题 | 传统对话方式的局限 |
|------|-------------------|
| 会话中断 | 对话关闭后，所有信息丢失，需要重新开始 |
| 上下文膨胀 | 每轮对话累积，Token消耗持续增长 |
| 进度丢失 | 无法恢复上次未完成的工作 |
| 修改追溯 | 难以追踪修改历史和当前状态 |

**实现方案**

1. **状态文件存储**

将已收集的数据结构化为JSON文件，存储在项目根目录下：

```
{项目根目录}/memory/prd-state.json
```

2. **状态对象结构**

```json
{
  "meta": {
    "productName": "产品名称",
    "currentPhase": 3,
    "progress": "2/5",
    "moduleProgress": {
      "total": 3,
      "completed": 1,
      "currentModule": "订单管理"
    },
    "lastUpdated": "2024-01-15T10:30:00Z"
  },
  "data": {
    "product": { ... },
    "roles": [ ... ],
    "modules": [ ... ],
    "flows": null,
    "nonFunctional": null
  }
}
```

3. **持久化时机**

| 时机 | 操作 | 说明 |
|------|------|------|
| 阶段完成 | Write保存状态 | 每完成一个阶段立即保存 |
| 模块完成 | Write保存状态 | 功能模块阶段每完成一个模块保存（细粒度） |
| 修改确认 | Write更新状态 | 用户确认修改后更新对应字段 |
| 输出生成 | Write保存完整状态 | 生成文档前保存最终状态 |

4. **状态恢复流程**

```
启动skill
    ↓
Read读取 prd-state.json
    ↓
文件存在？
    ├── 否 → 创建初始状态，开始新会话
    └── 是 → 检查进度
              ├── 已完成 → 询问是否开始新需求
              └── 未完成 → 询问是否继续上次进度
                           ├── 继续 → 恢复状态，继续收集
                           └── 重新开始 → 清空状态，开始新会话
```

**配合摘要策略**

记忆机制与摘要策略结合，实现上下文压缩：

```
传统方式（无记忆机制）：
┌─────────────────────────────────────┐
│ 完整对话历史（持续增长）              │
│ - 阶段1所有问答                      │
│ - 阶段2所有问答                      │
│ - 阶段3所有问答                      │
│ ...                                  │
└─────────────────────────────────────┘

记忆机制 + 摘要策略：
┌─────────────────────────────────────┐
│ 状态文件（prd-state.json）           │
│ - 已收集的结构化数据                  │
│ - 当前进度信息                       │
├─────────────────────────────────────┤
│ 对话上下文（仅当前阶段）              │
│ - 当前阶段的问答                     │
│ - 待确认的问题                       │
└─────────────────────────────────────┘
```

**效果**

| 指标 | 无记忆机制 | 有记忆机制 | 改进 |
|------|-----------|-----------|------|
| 会话恢复 | ❌ 不支持 | ✅ 支持跨会话恢复 | 中断后可继续 |
| 上下文大小 | 持续增长 | 仅当前阶段 | 主会话Token消耗减少50%+ |
| 进度追踪 | ❌ 不直观 | ✅ 可视化进度 | 随时了解完成情况 |
| 数据完整性 | ❌ 依赖对话历史 | ✅ 结构化存储 | 不丢失任何信息 |

**与上下文压缩的协同**

记忆机制是上下文压缩的基础设施：

1. **状态文件作为压缩后的摘要**：所有已收集信息以结构化形式存储
2. **/clear提示实现真正的上下文清理**：清空对话历史，状态文件保留
3. **阶段切换无缝衔接**：恢复状态后直接进入下一阶段



### 上下文压缩

**目的**

1. **提升模型性能**：研究表明，LLM在长上下文中的性能会显著下降，尤其是需要从上下文中检索信息时，"中间位置"的信息最容易被忽略（Lost in the Middle现象）

2. **降低Token成本**：减少每次请求的Token数量，直接降低API调用成本

3. **提高响应速度**：更少的上下文意味着更快的推理速度


**已有的实现方法**

**方法1：滑动窗口压缩**

只保留最近N轮对话，更早的对话通过摘要保留关键信息：

```
┌─────────────────────────────────────┐
│ 摘要区（固定保留）                    │
│ - 产品名称、核心描述                  │
│ - 已完成的阶段状态                    │
│ - 关键决策记录                       │
├─────────────────────────────────────┤
│ 活跃区（滑动窗口，保留最近3-5轮）      │
│ - 当前阶段的对话                     │
│ - 待确认的问题                       │
└─────────────────────────────────────┘
```

**方法2：分层压缩策略**

| 层级 | 内容类型 | 压缩方式 | 保留比例 |
|------|---------|---------|---------|
| L1 | 状态对象 | JSON结构化存储 | 100% |
| L2 | 当前阶段对话 | 原样保留 | 100% |
| L3 | 已完成阶段 | 摘要压缩 | ~20% |
| L4 | 历史迭代 | 仅保留最终结果 | ~5% |

**实际使用方法**

由于 / 开头斜杠命令是 ClaudeCode 系统内置的，不能通过自定义 tool 来拓展。因此，考虑增加阶段提示，在任务的不同阶段设置提示，告诉用户手动调用 /clear + /skill。同时，在skill中增加启动流程，进入skill时先检查记忆状态，提示当前进度。



### 任务拆分

**问题**

完整的模块功能会比较复杂，在此基础上生成的需求文档和html原型图更复杂。

| 问题     | 具体表现                          |
| -------- | --------------------------------- |
| 输出截断 | 文档+原型内容过长，单次生成不完整 |
| 质量不稳 | 复杂结构难以一次正确组织          |
| 内容遗漏 | 多章节容易丢失某些部分            |

案例：10K词左右的功能清单已经存在大量遗漏，实际需求中的业务内容一般都会比较复杂。

**解决方法**

子agent + checklist

**方案对比：TaskCreate vs Agent**

| 对比项     | TaskCreate（任务列表）    | Agent（子Agent）      |
| ---------- | ------------------------- | ----------------------|
| 上下文 | 同一上下文 | 独立上下文 |
| 执行方式 | 顺序执行（由主对话控制） | 可并行执行 |
| 数据共享   | 直接共享（同一对话）      | 需要通过结果传递      |
| 用途       | 任务进度跟踪              | 独立子任务执行       |
| Token消耗  | 在主上下文中              | 独立上下文，互不影响 |

**实现方案**

1. 主agent开启多个agent创建任务分模块生成需求
2. 主agent在子agent生成完后对比prd-state.json中的原始数据和子agent输出内容，检查模块化内容是否一致。
